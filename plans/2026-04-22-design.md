# glimpse-rs — Design Plan

Date: 2026-04-22

## Purpose

A small Rust service for running a private family blog. Content is primarily
photo-based with some text per post setting context. Used for sharing vacation
updates, family gatherings, and updates on the kids with close family and
friends.

Goals:

- Simple, beautiful design with photos as the centerpiece.
- Content authored as Hugo-style folders: a markdown file with frontmatter
  plus a `photos/` subfolder.
- Per-post access control by groups (e.g. `family`, `friends`, `public`).
- Minimal JavaScript; server-side HTML rendering.
- Separation of content parsing from HTML/CSS rendering so themes can be
  swapped.

## Non-goals (for now)

- CDN integration. Majority of posts will be restricted, so edge caching
  doesn't pay off. Revisit later if needed.
- Per-photo (sub-post) access control.
- Comments, reactions, or any user-generated content.
- Full-text search.

## Content model

A post is a folder under `posts/`:

```
posts/
  2025-03-18 Hawaii/
    index.md
    photos/
      2025-03-18 Travel day/
        *.jpg
      2025-03-19 Manoa Falls hiking/
        *.jpg
      ...
```

`index.md` has YAML frontmatter plus markdown body:

```yaml
---
title: "2025-03 Hawaii"
date: 2025-03-18
access: [family]          # groups allowed; omit = draft (admin-only)
cover: "photos/.../x.jpg" # optional hero image
---
```

- `access` is a list of group names. A post is visible if the viewer's
  groups intersect this list, the post explicitly lists `public`, or the
  viewer is in the `admin` group.
- **Omitting `access` (or leaving it empty) makes the post a draft** —
  visible only to `admin`. This is the default so that new post folders
  committed before their audience is decided are never accidentally
  exposed. Publishing a post is an explicit act: adding at least one
  group.
- No separate `draft: true` flag is needed; the presence/absence of
  `access` is the single source of truth.
- When an admin views a draft (in the index list or on the post page
  itself), the default theme renders a visible **DRAFT** badge so the
  state is unambiguous.
- Photos may be flat under `photos/` or organized into subfolders. Subfolder
  names become section headings and produce one gallery per subfolder,
  rendered in the position of a `{{< gallery >}}` shortcode (or appended if
  absent).

## Users & groups

File-backed config to start — no database. Password resets are handled
manually by an admin editing `users.toml` (replacing the `password_hash`
field). Fine at family scale; revisit only if it becomes a burden.

```toml
# users.toml
[[users]]
username = "grandma"
password_hash = "..."   # argon2
groups = ["family"]
```

Groups are just strings. The `admin` group bypasses access checks.

Future option: signed share links per post for non-account viewers. Not in
the initial build.

## Rendering strategy — fully dynamic (Option C)

HTML is rendered per request from an in-memory `Site` model. No `public/`
output directory.

Rationale: post pages may eventually carry per-viewer content (e.g., "new
since your last visit", draft previews for admins on the live URL, signed
share-link elevation). Pre-materializing HTML would be wasted work and a
future migration.

Key points:

- Markdown is parsed **once** at site-load time; rendered HTML fragments
  are stored on the `Post` struct. Per-request cost is template
  composition only (sub-ms with MiniJinja at this scale).
- Photos (the expensive bytes) are served as plain static files from disk,
  gated by the parent post's access rules.
- Thumbnails and intermediate sizes are generated into a cache dir with
  content-hashed filenames; idempotent and cheap to regenerate.

Access enforcement lives in one function used by every route:

```rust
fn visible<'a>(site: &'a Site, viewer: &Viewer) -> impl Iterator<Item = &'a Post>
```

The media route checks the parent post's access before streaming the file.

## Architecture

Modules with clear boundaries so themes and auth evolve independently:

1. **`content`** — scans `posts/`, parses frontmatter + markdown, discovers
   photos, produces a `Site { posts: Vec<Post> }` model. Pure, no HTML, no
   HTTP.
2. **`media`** — generates derivative images (thumbnail, medium, full) into
   a cache dir; reads EXIF for date/orientation.
3. **`theme`** — takes the `Site` model + a theme directory (templates +
   CSS) and exposes render functions:
   - `render_index(&Site, &Viewer) -> String`
   - `render_post(&Site, &Post, &Viewer) -> String`
   - `render_login(...) -> String`
   The theme never sees an HTTP request.
4. **`server`** — Axum app that routes requests, authenticates, enforces
   access, and calls the theme. Owns sessions, cookies, and static-file
   serving.

### Request routing

```
GET /                 → filter Site by viewer.groups, render_index
GET /posts/<slug>     → load post, check access, render_post
GET /media/<post>/*   → access check on parent post, ServeDir on cache/
GET /static/*         → theme assets (CSS, fonts, small JS if any)
POST /login, /logout  → sessions
```

### State

```rust
struct AppState {
    site: Arc<ArcSwap<Site>>,   // lock-free reads; swapped on rebuild
    theme: Arc<Theme>,          // compiled templates
    users: Arc<Users>,
    media_cache: PathBuf,
}
```

File-watcher (`notify`) rebuilds `Site` off-thread and atomically swaps it
in; no request ever sees a half-built state.

## RSS / Atom feeds

Feeds are personalized by access group using per-user tokenized URLs. This
is the standard pattern for private feeds (Feedbin, Miniflux, etc.) and
works with any feed reader that fetches plain HTTPS.

### URL shape

```
GET /feed/<token>.xml
```

- Token is 128+ bits of entropy (base64url), minted per user.
- Stored **hashed** in `users.toml` alongside the password hash, so a
  leaked config file doesn't hand out live feed URLs.
- One active token per user initially; rotatable by admin action to
  revoke a reader (e.g., lost device).
- Unknown token → 404 (not 401), to avoid confirming the endpoint
  exists.

### Rendering

Add a render function to the `theme` module:

```rust
fn render_feed(&Site, &Viewer) -> String
```

The route handler:

1. Looks up the token → `Viewer` (or 404).
2. Filters `site.posts` via the same `visible(site, viewer)` used by
   the index.
3. Renders Atom via the `atom_syndication` crate (Atom handles
   Unicode, dates, and relative links better than RSS 2.0).
4. Sets `Cache-Control: private, no-store` — the token is the
   credential.
5. Emits `<updated>` timestamps so readers can conditionally fetch
   with `If-Modified-Since`.

### Images in feeds

Private feeds need to render images without a session cookie, because
feed readers generally won't carry cookies to embedded `<img>` URLs.

- Feed HTML embeds the `medium` derivative (not `full`) inline in
  `<content type="html">`.
- Image URLs in the feed are signed with the same token:
  `/media/<post>/<hash>.jpg?t=<token>`.
- The media route accepts either a valid session cookie **or** a valid
  `?t=` token as authentication; either must grant access to the
  parent post.

### Stack additions

- **atom_syndication** — feed rendering
- **rand** — token generation (already likely pulled in by other deps)

### Non-goals for feeds

- Per-device tokens (one per user is enough to start; revisit if
  someone actually needs to revoke one device without the others).
- One-feed-per-group unauthenticated URLs — rejected because
  secret-URL sharing is weaker than token-per-user and doesn't
  compose for multi-group users.

## Directory layout

```
glimpse-rs/
  src/
    content/    parser, site model
    media/      thumbnailer, cache
    theme/      renderer
    server/     axum, auth, sessions
    config.rs
    main.rs
  themes/
    default/    templates/, static/ (css)
  posts/        user content
  cache/        generated thumbnails
  users.toml
  config.toml
```

## Stack

- **axum** — HTTP
- **tower-sessions** — cookie sessions
- **tower-http** — static file serving, compression
- **minijinja** — templates (lighter than Tera, fine for this scale)
- **pulldown-cmark** — markdown
- **gray_matter** or **serde_yaml** — frontmatter parsing
- **image** + **kamadak-exif** — thumbnails, EXIF
- **notify** — file-watching
- **argon2** — password hashing
- **arc-swap** — lock-free `Site` hot reload
- **serde**, **toml**
